1. System Overview

Salon Manager is a multi-tenant business management platform for salon and retail outlets. A single deployment currently serves two Naturals salons and one FirstCry store under one company, sharing one billing system and one loyalty programme ("Gems").

It is a serverless application: the backend is a Google Apps Script (GAS) Web App that stores all data in a single Google Spreadsheet; the frontend is a static single-page application (SPA) hosted on GitHub Pages. There is no traditional database, application server, or container — the spreadsheet is the database and GAS is the API tier.

The product covers: point-of-sale billing, service & product catalogues, price books, inventory with purchase orders / stock receipts / audits, vendors, a four-tier loyalty programme, appointments, expenses, a full HR suite (staff, shifts, weekly schedules, attendance, advances, incentive profiles, payroll), organisations, users, roles and permissions. Two additional standalone entry points — a customer loyalty portal and a staff self-service portal — reuse the same backend.

  • One endpoint, many actions. Every request is an HTTP POST to the GAS Web App URL with a JSON body { action, ...data, sessionToken }. Main.js switches on action and dispatches to a module method.
  • Flat JSON responses. Every handler returns Utils.createResponse(status, message, payload) which serialises { status, message, ...payload } — the payload is spread at the top level, never nested.
  • No CORS headers & no custom request headers. Requests are sent without a Content-Type header to avoid a CORS pre-flight against GAS; responses set only the JSON MIME type. This is why the frontend never sets headers on fetch.
  • No doGet — the backend is POST-only.

4. Multi-Tenancy Model

Tenancy is enforced by an orgId column present on almost every sheet, plus an organisation tree.

  • Org tree. The Organizations sheet has a self-referencing parentId. A blank parentId means a top-level org. The two Naturals outlets and FirstCry are child orgs under one parent company.
  • Scoping primitive. Organizations.getOrgAndChildren(orgId, allOrgs) recursively returns an org plus all descendants. A parent-org user sees their whole branch; a leaf-org user sees only their outlet.
  • Row filtering rule (used by every getAll): a row is included when !orgId || !rowOrg || rowOrg === orgId. Rows with a blank org are never filtered out (legacy/shared data).
  • orgId source of truth. For admin actions, orgId and userId are injected server-side from the validated session (Main.js overwrites any client-supplied values). Client cannot spoof its org.
  • Never rewritten on update. orgId is written only at record-creation time; no module rewrites it on update.

5. Authentication & Sessions

5.1 Password hashing

Utils.hashPassword(pw) = SHA-256, unsalted, no iteration, returned as lowercase hex (Utilities.computeDigest(SHA_256, pw)). Passwords are stored in Users column 2 and are never returned by any read.

5.2 Admin login

  1. Auth.login({email,password}) reads the Users sheet (9 columns).
  2. Match = email === row.email && hashPassword(password) === row.password.
  3. If status === 'active': fetch role permissions, create a session, return flat payload { sessionToken, userId, email, fullName, phone, whatsapp, orgId, roleId, permissions }.
  4. Inactive account → error; no match → Invalid email or password.

5.4 Password reset flow

  • requestPasswordReset(email) → generates a UUID token, expiry = now + 1 hour, appends [token, email, expiry, false] to PasswordResetTokens, emails a link to reset-password.html?token=… via MailApp.
  • validateResetToken(token) → checks token exists, not used, not expired; returns {email}.
  • resetPassword(token, newPassword) → intended to hash & store the new password and mark the token used. defect — see §18.

6. Request Routing (Main.js)

doPost(e) parses JSON.parse(e.postData.contents), reads data.action, and dispatches through two switch blocks wrapped in a try/catch (any thrown error → error response):

  1. Staff switch (checked first). If the action is in STAFF_ACTIONS and is not staff_login, it validates the sp_ staff session (else returns {staffSessionExpired:true}) and injects staffId + orgId. All 10 staff actions return here and never fall through.
  2. Admin switch. If the action is not in publicActions, it validates the admin session (else "Session expired. Please login again.") and injects orgId + userId. Public actions skip validation but still route through this switch. Unknown action → Invalid action.

login, request_password_reset, validate_reset_token, reset_password, customer_login, get_customer_history, get_customer_loyalty, get_loyalty_ledger

10. Billing Engine (Bills.js)

10.1 save(data)

  1. Validates the Bills and BillItems sheets exist. No server-side validation of items/customer/totals — validation is client-side.
  2. IDs: bill = BILL + Date.now(); each line = BI + Date.now() + rand5. createdAt = full ISO timestamp.
  3. Splits items by type into service vs product and sums each side's lineSubtotal and lineGst (each Number()||0). GST percentages are not recomputed — the client's per-line lineGst is trusted and summed.
  4. Grand total = servicesSubtotal + servicesGst + retailSubtotal + retailGst − discount + tip. Here discount is the combined figure (manual discount + loyalty redemption value).
  5. Appends the Bills row (status hard-coded active) and one BillItems row per item.
  6. Calls _deductStock(billId, dateYYYYMMDD, items, userId, orgId).
  7. Calls LoyaltyPoints.processAfterBill(billId, customerPhone, customerName, pointsToEarn, redeemPoints, orgId) if a phone resolves (customerPhone || customerId).
  8. Returns { billId, grandTotal }.

10.2 _deductStock

For each retail product line, appends a StockMovements row of type billing with negative qty and decrements Products.currentStock. Professional-product consumption (profProductId + profQty) deducts at unit cost 0. Stock is allowed to go negative to stay in sync with the movement ledger. Clears the products cache.

10.3 voidBill / getAll / getItems

  • voidBill sets Bills.status = void. It does not reverse stock movements or loyalty ledger entries.
  • getAll defaults to the last 90 days; accepts fromDate/toDate (YYYY-MM-DD). Void bills are returned as-is (status included).
  • getItems returns all BillItems for a bill id.
Trust boundary The server recomputes bill totals from line items but trusts client-supplied lineGst, discount, pointsToEarn and redeemPoints. Redemption is deducted from the balance server-side only if the balance is sufficient.

11. Loyalty Engine (LoyaltyPoints.js)

11.1 The two-balance model

pointsBalance (Customers col 5)
The spendable wallet. Increased on earn, decreased on redeem. Persisted as balanceAfter in each ledger row.
statusPoints (Customers col 6)
The tier-qualification score. Never reduced by redemption. Fully recomputed as the sum of all non-expired earn entries in the rolling window. Determines the tier.

This is the airline model: spending Gems lowers your balance but never your tier.

11.2 Default config

{ enabled:false, pointsName:'Points', baseEarnRate:10,
  tiers:[ {name:'Tier 1',threshold:0,multiplier:1.00},
          {name:'Tier 2',threshold:500,multiplier:1.25},
          {name:'Tier 3',threshold:2000,multiplier:1.50},
          {name:'Tier 4',threshold:5000,multiplier:1.75} ],
  happyHourMultiplier:2.0, happyHourActive:false, happyHourSchedules:[],
  redemptionRate:100, redemptionValue:10, minRedemption:100, expiryMonths:12 }

Config is stored in OrgSettings['loyalty'] as JSON and merged over these defaults on load. The current programme names the currency Gems with four tiers and per-tier badge colours.

11.3 Earning

Points-to-earn are computed client-side in billing.js and trusted by the server (re-floored):

eligibleSpend = Σ lineSubtotal of rows whose service/product group is pointsEligible
pointsToEarn  = floor(eligibleSpend / 100) × baseEarnRate × tierMultiplier × happyHourMultiplier

Only groups flagged pointsEligible contribute. Happy-hour multiplier applies only when the server reports isHappyHour at lookup time.

11.4 Happy hour detection

  1. Manual override: if happyHourActive is true → active immediately.
  2. Else each schedule is checked: today's day must be in days; effectiveFrom/effectiveUntil window must include now; the current HH:MM must fall within startTime–endTime.

11.5 processAfterBill — order of operations

Returns immediately if the programme is disabled. Then, in order:

  1. Redeem first (only if balance ≥ redeem): balance −= redeem; append LPR… ledger row (type redeem, negative points, empty expiry); write new balance to Customers col 5. Insufficient balance is silently skipped (no partial redemption).
  2. Earn (earn = floor(pointsToEarn)): expiry = now + expiryMonths; balance += earn; append LPE… ledger row (type earn, positive points, expiry set); write balance.
  3. If anything changed, call _recalcStatus.

11.6 _recalcStatus — status points & tier

statusPoints = Σ points of every ledger row where type=earn, phone matches, earnedDate ≥ (now − expiryMonths) and expiryDate ≥ now. Then:

qualifiedIdx  = highest tier whose threshold ≤ statusPoints
currentIdx    = index of the customer's current tier
minAllowedIdx = max(0, currentIdx − 1)      // one-step downgrade protection
newTierIdx    = max(qualifiedIdx, minAllowedIdx)

So a customer can drop at most one tier per recalculation. statusPoints → Customers col 6, tier → col 7.

12. Inventory & Stock

  • Stock movements record every change: receipt (positive, from receiveStock), billing (negative, from a bill), audit (signed variance). The register is the running audit trail.
  • Receive stock increments currentStock and writes a receipt movement; if a poId is supplied it also bumps POItems.qtyReceived and rolls the PO status.
  • PO status (auto): draft on create → partial (some received) → received (all items fully received). Can also be set manually.
  • Stock audit: variance = physicalQty − systemQty; if non-zero, writes an audit movement and overwrites currentStock to the physical count.
  • Price resolution: a price book returns an explicit override (isDefault:false) if one exists, otherwise the service's defaultPrice (isDefault:true, itemId:null).
Not in codebase PO "smart suggest" (auto-fill items below reorder point) and any dedicated reorder logic are not implemented server-side. The enabling field baseStock (reorder level) exists on Products, but no backend function generates suggestions.

13. HR, Attendance & Payroll

13.1 Time handling

Google Sheets stores bare time cells as Date objects anchored to the 1899-12-30 epoch. _fmtTime() detects instanceof Date and formats only HH:mm in the script timezone; string inputs are regex-parsed for a leading HH:mm.

13.2 Overtime

otHours = max(0, hoursWorked − 9) — a flat 9-hour threshold, no break deduction (even though Shifts has a breakMins column). This rule is duplicated in Attendance.saveAttendance, HRApprovals._calcHours, and consumed by Payroll.

13.3 Attendance & approval lifecycle

  • dayStatus ∈ {present, absent, half-day}. Row status ∈ {pending, approved, rejected} (default approved for admin-entered records; staff-entered start pending).
  • Manager approves/rejects via HR Approvals; rejected records can be re-approved.

13.4 Advances — two paths

  • Admin direct (Attendance.addAdvance): inserted straight as disbursed.
  • Staff request: pending → approved (with approvedAmount) → disbursed (with paymentMode). Outstanding balance counts only disbursed rows.

13.5 Incentive profiles & tiered commission

A profile defines two revenue targets and three commission brackets. L1 = l1Type==='salary_pct' ? salary×l1Value/100 : l1Value (same for L2). Given monthly revenue R (org- or individual-based per revenueBase):

R < L1            → 0
L1 ≤ R < L2       → L1·X% + (R−L1)·Y%
R ≥ L2            → L1·X% + (L2−L1)·Y% + (R−L2)·Z%

13.6 Payroll calculation

Inputs: staffId, period (YYYY-MM), eligibleOffs (default 4), payableDays (default 26), advanceDeducted.

  • Weekend weighting (peak days = Fri/Sat/Sun): absent on a peak day counts as 2 days off (weekday = 1); half-day peak = 1 day (weekday = 0.5).
  • excessLeaves = max(0, totalDaysOff − eligibleOffs); leaveDeduction = excessLeaves × (salary / payableDays); adjustedBaseSalary = salary − leaveDeduction.
  • otPay = otHours × profile.otHourlyRate.
  • Incentives: computed monthly from bills (service revenue, product unit-incentives, direct-incentive "makeup", and the tiered target incentive).
totalIncentive = targetIncentive + directIncentive + productIncentive
netPay = adjustedBaseSalary + allowances + otPay + totalIncentive − advanceDeducted

calculate returns a breakdown (status draft) and does not persist; save appends the Payroll row.

14. Frontend Architecture

  • index.html is a shell — login screen, app frame (header/sidebar/main), empty content-section divs, and global modals.
  • app.js fetches all 16 pages/*.html fragments at load (_loadPageHTML()) and injects them into the matching sections, then runs each module's init() inside a try/catch.
  • Navigation.switchPage(page) toggles the active section and lazy-loads that module's data on first visit via _callLoad. A background preloader warms heavy pages 1.5 s after login.
  • Navigation.applyPermissions() hides home tiles for menu items where canAccess === false.
  • Multi-section pages use the prod-tabs pattern (tab button + panel).
  • api.js — one call(action, data) transport: attaches sessionToken from localStorage, POSTs JSON, handles the expired-session message globally, plus one typed wrapper per action.

Pages: dashboard, billing, history, appointments, expenses, services (3 tabs), products (6 tabs), vendors, staff (5 tabs), hrapprovals (2 tabs), customers, organizations, users, roles, permissions, settings. CSS load order: base → components → billing → pages → setup. (Legacy styles.css exists but is not linked.)

16. Setup & Verification

  • get_setup_status compares each schema against the live sheet by column count: absent → missing; fewer columns than expected → missing_columns (the missing tail); otherwise ok. It never flags header naming or order.
  • run_setup takes actions create (insert sheet + write canonical headers, bold + freeze row 1) or add_columns (append the missing tail). Creating IncentiveProfiles also seeds three default profiles. It always rebuilds the 📋 Index summary sheet afterward.
  • refresh_summary_sheet rebuilds the Index sheet — a formatted, hyperlinked table of every sheet grouped by category, with descriptions and column counts.

18. Known Issues & Design Caveats

Bug — password reset Auth.resetPassword checks validation.status, but validateResetToken returns a ContentService TextOutput (no .status property). The guard therefore mis-branches and the password-update path is effectively unreachable on success. Needs a fix to read the parsed content instead.
Data integrity Customer bill history matches by name (case-insensitive), not phone. A rename or a name collision would mis-attribute bills.
Reversal gaps Voiding a bill does not reverse stock movements or loyalty ledger entries.
  • Staff.update and every other update never rewrite orgId — org reassignment is unsupported.
  • minRedemption exists in config but is not enforced in billing.js.
  • Loyalty redemption is double-represented on save: as rupees folded into discount and as points in redeemPoints. Intentional, but worth noting.
  • WeeklySchedule's "current/future weeks only" edit rule is enforced in the UI, not the backend.
